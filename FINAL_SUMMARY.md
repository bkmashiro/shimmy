# Shimmy Sandbox 安全研究总结

**日期:** 2026-03-09  
**研究时长:** ~3小时  
**研究员:** 明石 (CTO)

---

## 1. 项目背景

Lambda Feedback (shimmy) 需要安全执行学生提交的Python代码。要求：
- 阻断网络访问（防止作弊/攻击）
- 限制资源使用（CPU/内存/文件）
- 允许Python完整运行（包括threading、numpy等）
- 低延迟（<5ms开销）
- Lambda环境兼容（无root、无KVM）

---

## 2. 技术方案选型

### 评估的三种方案

| 方案 | 隔离级别 | 延迟 | Lambda兼容 | Python支持 |
|------|----------|------|------------|------------|
| **sandbox_exec (用户态)** | 进程+seccomp | ~1.5ms | ✅ | ✅ 完整 |
| RootSandbox (namespace) | 容器级 | ~5ms | ❌ 需root | ✅ 完整 |
| WASMSandbox | 虚拟机级 | ~10-50ms | ✅ | ⚠️ 受限 |

### 选择: sandbox_exec (用户态seccomp)

**原因:**
1. Lambda不支持user namespace
2. 需要完整Python+C扩展
3. 延迟要求<5ms
4. Lambda已有容器隔离作为底层防护

## 3. 实现细节

### 3.1 技术栈

```c
// sandbox_exec.c (224行纯C)
├── seccomp-bpf     // syscall过滤 (62个阻断)
├── rlimit          // 资源限制
├── prctl           // NO_NEW_PRIVS, 禁止dumpable
├── setsid/setpgid  // 进程组隔离
└── fork+exec       // seccomp在exec前生效
```

| 类别 | Syscall |

| 进程 | clone(无THREAD), ptrace, process_vm_readv, process_vm_writev |
| 信号 | kill(-1), tkill, tgkill |
| 内核 | io_uring_*, bpf, userfaultfd, perf_event_open |
| 密钥 | keyctl, add_key, request_key |
| 监控 | inotify_*, fanotify_* |
| 文件 | mount, umount2, symlink, link, chroot, pivot_root, open_tree, move_mount, fs* |
| 系统 | reboot, kexec_*, init_module, finit_module, delete_module, acct, swap*, set*name, *time* |
| 硬件 | ioperm, iopl, modify_ldt |
| 其他 | personality, quotactl, nfsservctl |

## 5. 已知限制

### 5.1 无法在sandbox层面解决

| 限制 | 原因 | 缓解措施 |
|------|------|----------|
| /proc可读 | 需mount namespace | --clean-env减少泄露 |
| /dev/shm持久化 | 需tmpfs挂载 | shimmy每次清理 |
| NPROC每用户 | Linux设计 | Lambda容器级限制 |

### 5.2 信息泄露 (低风险)

```
/proc/self/environ  → --clean-env缓解
/proc/self/maps     → ASLR信息，但无法利用
/proc/1/environ     → 可能含配置，shimmy需清理
/proc/net/tcp       → 网络信息，但无法建连
```

## 8. 部署建议

```go

    "--no-fork", "--no-network", "--clean-env",
```

## 10. 结论

sandbox_exec是Lambda环境下的最优解：
- 224行纯C，无外部依赖
- 62个危险syscall阻断
- 60个威胁测试100%通过
- ~1.5ms启动开销
- 5轮红队对抗验证

**安全不是绝对的，但对于学生作业评测场景，这个方案提供了足够的防护。**
